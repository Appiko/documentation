6 # SensePi Firmware
7 
8 [SensePi](https://appiko.org/sensepi.html) is a motion based camera triggering device. This device can be used
9 to trigger DSLRs and Point and Shoot cameras. This device can be used as a trigger
10 for other systems as well.
11 
12 There is no active sensor being used in this device.
13 
14 
15 
16 ## Application Explanation :
17 
18 SensePi Firmware is an application which handles multiple modules which serves
19 a different purpose like handling camera triggering, wireless connection, data
20 storage into non-volatile memory.
21 
22 This application also handles events generated by some sub-modules like
23 button press module, add tick module, irq msg handler, etc.
24 
25 ### Working Principle :
26 
27 SensePi Firmware works on event-driven principle. That means core will wake-up
28 only when some interrupt or events happens. Other times it'll be in sleep mode.
29 This will result in a drastic drop in power consumption. Also by making it
30 event-driven its response latency is very low. There are almost no instructions
31 to perform between event and handler operations. The response time of the device
32 is very low. Control diagram will explain how program's flow is.
33 
34 Firmware's working can be explained as a state machine having 3 states.
35 The application will be always in one these states. These states are:
36 
37  - Advertisement
38  - Connected
39  - Sensing
40 
41 This state machine mechanism is handled by following sub-modules :
42 
43 #### 1 IRQ message module :
44 The state machine mechanism is handled by application level events. These events are
45 Add Tick Event and State Change Event. These events are triggered by interrupts of
46 different modules. To handle these events we have used the circular buffer. These
47 events are treated as messages in the buffer. In interrupt handler of respective modules,
48 these events are pushed into the circular buffer as a message. These messages then
49 popped in the main thread and events will be handled.
50 
51 #### 2 System tick module :
52 
53 This module generates Add Tick Event after a certain number of ms_timer_ticks has passed,
54 say 'n' ticks. This module uses ms_timer to generate an event. Every time the processor
55 wakes up, this module checks how many ms_timer_ticks have passed since last event occurrence,
56 let's say it is 'tick count'. If that 'tick count' is greater than that of 'n/2' it generates
57 Add Tick event and starts counting ms_timer_ticks again. If 'tick count' is less than 'n/2'
58 module will not do anything. If the processor doesn't wake up for 'n' ms_timer_ticks, this
59 module will generate Add Tick event. The value of 'n' can be changed. This value will decide
60 maximum sleep time at any instance of time.
61 
62 #### 3 Button Press module :
63 
64 Button press module works on button press duration. This sub-module will come into
65 play when we press the button. From that instance, it'll calculate how long the button
66 has been pressed. Here we have divide button press durations in 3 different 'Durations':
67 Pressed, Short and Long. As soon as the user presses the button, system ticks duration is
68 changed to 100ms(in form of ms_timer_ticks here onwards). At every Add Tick Event, it
69 adds 100ms to count and checks at which 'Duration' module is. At 'short press duration',
70 the module will change SensePi's state to advertisement state. At 'long press duration' the
71 module forces SensePi into bootloader mode for a firmware update.
72 
73 ### States Machine :
74 In each state, different sets of peripheral modules are active or serving different
75 requests. In SensePi we use BLE stack for Bluetooth connectivity.
76 
77 #### 1 Advertisement :
78 
79 In this state, device's Bluetooth is on and ready to connect. To understand working
80 of Advertisement state with respect to BLE stack read this [article](https://devzone.nordicsemi.com/b/blog/posts/bluetooth-smart-and-the-nordics-softdevices-part-1).
81 In advertisement device will transmit specified data periodically. Then user can
82 connect to device, possibly using 'Appiko setup' app or 'nrf Connect' app.
83 Timeout for this state is 5 mins.
84 
85 #### 2 Connected :
86 
87 In this state, device is connected to a mobile device and hence cannot connect
88 with other mobile devices. To understand working of Connected state with respect
89 to BLE protocol read this [article](https://devzone.nordicsemi.com/b/blog/posts/bluetooth-smart-and-the-nordics-softdevices-part-2).
90 A user can send configurations from mobile to the device. Timeout for this state is
91 10 mins.
92 
93 #### 3 Sensing :
94 
95 This is the default state for the device. After Timeout or after disconnect SensePi
96 will comeback in this state. This state handles PIR sensing & camera triggering.
97 There is no timeout for this state.
98 
99 By using this state machine approach we were able to avoid a lot of conflicting
100 scenarios
101

103 
104 The state diagram for SensePi is given below:

108 As one can see, the default state when SensePi starts it's operation is Sensing.
109 If someone presses button present on SensePi, it'll go into Advertising state.
110 There it'll wit for 3 mins for the Connection request. If it receives a connection request
111 before 3 mins over, it'll connect with the requesting device.
112 
113 Once SensePi connects with some device, it can remain in this state for 10 mins.
114 After 10 mins timeout will occur and it'll go back to Sensing state. If a user have
115 proper mobile device and proper application to change the configuration, a user can
116 do it in these 10 mins. And a user can also terminate the connection by sending disconnect
117 request.

119 ### Data Flow Diagram :
120 
121 This State machine implementation is done by building a firmware application in a
122 modular structure. There are 3 supporting modules which handles different aspects
123 of firmware. Those modules are :
124 
125  1. Bluetooth communication module
126  1. Camera triggering module
127  1. Data storage module
128 
129 An interactions between these modules and application can be represented as follow :
130 
131 ![](./doc/data_flow.png)
132 
133 In the context of this diagram communication means data transfer between any
134 two modules. Here all the arrows represents a transfer of sensepi_config until specified
135 otherwise. Data transfer happens bidirectionally between modules, except for
136 2 cases where it's unidirectional.
137 
138 We can observe over there, all modules interact with the main application directly.
139 Almost all the inter-module communication is done through the main application. Only
140 once direct communication between two modules happens when Camera module saves
141 latest configuration in Flash memory through Data storage module. Passing of new
142 configuration from Bluetooth communication module to camera trigger module is also
143 done through main application.
144 
145 On system power reset or firmware update, main application will load the latest
146 configuration present in flash memory through Data storage module and will pass it
147 on to Camera trigger module.
148 
149 ### Different Modules of application
150 #### 1 Bluetooth communication module
151 
152 This module handles Bluetooth communication. i.e. this module decides what kind of
153 data is to be sent over Bluetooth. The structure which is to be used to send the
154 data over Bluetooth is defined in this module. This module also handles how to
155 serve different requests that it might get while using Bluetooth to communicate
156 with other devices. One can refer to data structures over here.(link)
157 
158 #### 2 Camera triggering module
159 
160 This module handles the operation of a device when it's in sensing state. This module
161 handles the camera triggering related operations. This module uses the data sent by
162 a mobile device to the Bluetooth communication module and does the required configurations.
163  These configurations contain mode of operation (i.e. timer, motion or both),
164  light conditions (i.e day, night or both), type of trigger (different modes in
165 which cameras can be triggered).
166 
167 #### 3 Data storage Module
168 
169 This module handles the storage if data such as current config and firmware id in
170 permanent memory. This data enables the device to restore the configuration data after
171 OTA update is done or if the power supply gets disturbed. This module uses flash memory
172 to store the data
